# AI 项目开发补充指南

本文件是 `AI_SYSTEM_RULES.md` 的项目级补充说明，用于约束当前项目的既有目录、公共样式、公共区域和发布边界。

使用方式：每次开发时，先让 AI 完整读取 `AI_SYSTEM_RULES.md`，再读取本文件，最后读取本次需求相关的 ROOT_MAP / TREE_MAP 和源码文件。

---

## 1. 文件定位

`AI_SYSTEM_RULES.md` 负责通用规则：

- ROOT_MAP / TREE_MAP 的生成、拆分和同步。
- 文件系统索引流程。
- 300 行限制。
- 职责拆分规则。
- 修改前后的强制执行流程。

本文件只负责当前项目补充约束：

- 当前项目应沿用既有目录结构。
- 公共样式、公共脚本、公共模板和公共区域不得走形。
- 前端、后端、本地构建器和发布产物不得混改。
- SEO、表单、接口、上传、弹窗、历史任务等敏感行为不得被悄悄改变。

如果本文件和 `AI_SYSTEM_RULES.md` 冲突，以 `AI_SYSTEM_RULES.md` 为准。

---

## 2. 目录结构原则

当前项目应以真实文件系统和 ROOT_MAP / TREE_MAP 为唯一目录依据。

AI 不得因为通用经验，擅自新增、迁移或推荐以下模板化目录：

```text
src/
src/assets/
assets/css/
assets/js/
partials/
components/
```

如果现有项目已经使用这些目录，可以继续沿用；如果项目没有这些目录，不得为了“规范化”强行创建。

确实需要调整目录结构时，必须先说明：

- 当前结构为什么无法满足需求。
- 会影响哪些构建、预览、发布或部署流程。
- 哪些文件需要新增、移动、删除或改名。
- ROOT_MAP / TREE_MAP 需要如何同步更新。
- 如何验证迁移后页面、交互、接口和构建行为不变。

没有用户确认时，不做目录迁移。

---

## 3. 公共样式约束

当前项目如果存在公共样式文件、全局样式文件、主题变量文件、布局样式文件或复用区块样式文件，AI 必须把它们视为高影响范围文件。

修改公共样式前必须先判断：

- 是否会影响多个页面。
- 是否会影响移动端布局。
- 是否会影响弹窗、表单、上传区、历史任务、导航、页脚等公共区域。
- 是否存在页面私有样式可以承接本次修改。
- 是否需要在 TREE_MAP 中标注该文件的影响范围。

优先级：

1. 只影响单个页面的样式，优先写入页面私有样式文件。
2. 多个页面复用且语义一致的样式，才允许进入公共样式文件。
3. 不得为了一个页面的临时效果污染公共样式。
4. 不得随意修改已有 class、id、data-* 选择器，除非确认没有脚本、模板或外部样式依赖。

公共样式文件在 ROOT_MAP / TREE_MAP 中必须明确标注“影响全站样式”或“影响多个页面样式”。

---

## 4. 公共区域和公共模板约束

当前项目如果存在公共头部、导航、页脚、侧边栏、弹窗、上传区、历史任务、API 配置区、语言切换区、公共卡片、公共按钮或其他复用区域，AI 必须优先判断本次修改属于：

- 公共区域修改。
- 页面私有区域修改。
- 公共组件样式修改。
- 页面私有样式或脚本修改。

修改公共区域前必须说明影响范围。

不得在没有确认的情况下：

- 改变公共区域 DOM 层级。
- 改变公共区域已有 class、id、data-*。
- 改变表单字段名。
- 改变按钮语义或跳转目标。
- 删除兼容性结构。
- 把页面私有内容写入公共模板。

公共区域文件在 TREE_MAP 中必须写清楚它控制哪些页面或哪些公共 UI 区域。

---

## 5. 功能链路保护

当前项目除 ROOT_MAP / TREE_MAP 外，必须同步维护 /docs/flow-root 与 /docs/flow-tree。

ROOT_MAP / TREE_MAP 负责回答“文件在哪里、文件干什么”。

FLOW_ROOT_MAP / FLOW_TREE 负责回答“用户动作会触发哪些代码、接口、数据、状态和关联功能”。

任何涉及页面交互、按钮、表单、链接、弹窗、上传、下载、登录、注册、保存、删除、API、数据库、状态、后台操作的修改，都必须先查功能链路索引，再修改代码，改完同步功能链路索引。

---

## 6. 前端修改约束

修改前端时，AI 必须先确认目标文件类型：

- 页面源文件。
- 公共模板或公共区域。
- 公共样式。
- 页面私有样式。
- 公共静态脚本。
- 页面私有脚本。
- 构建产物。

默认禁止直接修改构建产物，除非项目没有源文件，或项目文档明确说明当前只能修改产物。

前端修改不得悄悄改变：

- 页面整体视觉风格。
- 移动端布局。
- 弹窗行为。
- 上传流程。
- 历史任务展示。
- API 配置入口。
- 表单字段、参数名和提交路径。
- 页面跳转和按钮语义。
- 中英文页面之间的体验一致性。

如果页面存在多语言版本，修改一个语言版本时必须检查是否需要同步另一个语言版本。

### 6.1 跨页面跳转新标签页打开规则（强制）

**核心原则：所有指向站内其他功能页面的按钮/链接，默认必须以新标签页（`target="_blank"`）打开，不得覆盖当前页面。**

原因：
- **SEO 与用户停留时长**：用户在首页点击功能入口后，新标签页承载功能页面，原首页保持在后台未关闭状态。Google 等搜索引擎会认为用户持续停留在网站中，有利于提升网站质量评分。
- **用户体验**：用户可以同时浏览首页和功能页面，无需反复前进/后退，降低跳出率。

具体规则：

1. **功能入口按钮**：首页、导航栏、侧边栏等区域中指向具体功能页面的按钮（如"立即使用"、"进入工作台"、"开始修图"等），必须使用 `window.open(url, '_blank')` 或 `<a target="_blank" rel="noopener">` 在新标签页打开。

2. **无需新标签页的场景**：
   - 登录/注册等认证流程页面（用户通常需要在完成后自动跳回）。
   - 当前页面内的锚点跳转。
   - 同页面内的 Tab 切换、弹窗、折叠面板等交互。

3. **安全属性**：`<a>` 标签必须同时添加 `rel="noopener"` 防止 `window.opener` 安全风险；`window.open()` 方式已默认隔离，无需额外处理。

4. **多语言同步**：修改一个语言版本的跳转逻辑时，必须同步修改所有语言版本的对应按钮。

代码示例：

```html
<!-- 推荐方式一：<a> 标签 -->
<a href="/zh-cn/ai-image-studio/" target="_blank" rel="noopener">进入工作台</a>

<!-- 推荐方式二：<button> 使用 onclick -->
<button onclick="window.open('/zh-cn/ai-image-studio/', '_blank')">立即使用</button>

<!-- 错误方式（禁止） -->
<button onclick="location.href='/zh-cn/ai-image-studio/'">立即使用</button>
<a href="/zh-cn/ai-image-studio/">进入工作台</a>
```

## 7. SEO 和发布边界

没有用户明确要求时，不得修改：

- title。
- description。
- H1。
- canonical。
- hreflang。
- sitemap。
- robots。
- 结构化数据。
- 统计代码。
- 广告代码。

如果需求涉及 SEO，AI 必须先说明会影响哪些页面和搜索展示字段，再修改。

不得把开发调试信息、真实密钥、余额、价格、任务 ID、私有接口地址或临时图片地址写入前端、文档或提交文件。

---

## 8. 后端和构建边界

当前项目如果同时存在本地后台、构建器、前台页面和业务后端，AI 必须区分它们的职责。

不得把以下逻辑混在同一个文件或同一层目录中：

- 本地构建逻辑。
- 前台页面交互。
- 业务后端接口。
- 配置读取。
- 数据处理工具函数。
- 第三方服务调用。

后端文件变大时，应根据 `AI_SYSTEM_RULES.md` 按接口职责、配置职责、数据处理职责或第三方服务职责拆分。

不得硬编码真实 API Key、Token、Secret、账号密码或生产环境敏感地址。

---

## 9. 文件行数和职责拆分

当前项目执行 `AI_SYSTEM_RULES.md` 的 300 行限制。

拆分文件时，优先按职责拆分，不按行数机械切割。常见拆分维度：

- 页面。
- 页面区块。
- 公共模板。
- 公共样式。
- 页面私有样式。
- 页面私有交互。
- 工具函数。
- API 调用。
- 配置读取。
- 数据处理。

拆分后必须保持原有调用关系清晰，不得引入重复功能、隐藏副作用或跨模块循环依赖。

---

## 10. 修改前影响分析

每次修改前，AI 必须在心中完成并在需要时向用户说明：

- 本次需求影响哪些文件。
- 是否影响公共样式。
- 是否影响公共区域或公共模板。
- 是否影响页面结构。
- 是否影响 JavaScript 行为。
- 是否影响 SEO。
- 是否影响表单、上传、弹窗、历史任务、API 配置或跳转。
- 是否影响本地后台、构建器或业务后端。
- 是否需要同步更新 ROOT_MAP / TREE_MAP。
- 是否需要先拆分过大或职责过多的文件。

如果影响范围较大，应先给出方案和风险，再修改。

---

## 11. 禁止默认执行的行为

没有用户明确要求时，AI 不得擅自：

- 重写整个页面。
- 引入 React、Vue、Next、Nuxt 等前端框架。
- 引入大型依赖。
- 改变现有目录结构。
- 手动修改第三方依赖目录。
- 手动修改构建产物。
- 删除兼容性代码。
- 改变现有 class、id、data-*。
- 改变 SEO 字段。
- 改变表单字段名、接口参数或接口路径。
- 硬编码真实密钥、余额、价格、任务 ID 或图片地址。
- 把简单问题复杂化。

确实必须执行以上操作时，必须先说明原因、影响范围和验证方式。

---

## 12. ROOT_MAP / TREE_MAP 记录要求

生成或更新目录索引时，除遵守 `AI_SYSTEM_RULES.md` 外，还必须特别标注以下高影响文件：

- 公共样式文件。
- 全局变量或主题文件。
- 公共模板文件。
- 公共头部、导航、页脚、侧边栏文件。
- 弹窗、上传、历史任务、API 配置等复用交互区域。
- 路由、入口、构建配置、部署配置。
- 后端接口入口、密钥配置读取、第三方服务封装。

标注示例：

```text
/public/css/common.css —— 全站公共样式文件，影响多个页面布局与视觉，修改前必须确认页面私有样式无法满足需求
/templates/header.html —— 公共头部模板，控制全站导航结构与语言入口，修改影响所有引用页面
/static/js/upload.js —— 上传区公共交互脚本，影响上传流程、状态提示和历史任务联动
```

---

## 13. 修改后说明格式

每次完成开发后，AI 回复应包含：

```text
本次改动：
1. 修改了哪些文件
2. 每个文件改了什么
3. 是否影响公共样式或公共区域
4. 是否改变页面效果
5. 是否改变交互逻辑
6. 是否改变接口、SEO 或构建行为
7. 是否已同步 ROOT_MAP / TREE_MAP
8. 已运行哪些检查
9. 剩余风险或建议人工检查点
```

如果某些检查无法运行，必须说明原因。

---

## 14. 最重要的一句话

`AI_SYSTEM_RULES.md` 负责“怎么定位、怎么拆分、怎么同步索引”；本文件负责“当前项目哪些东西不能走形”。开发时应沿用真实目录，保护公共样式和公共区域，小步修改，改完同步地图。
